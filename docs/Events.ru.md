# Класс Events

## Назначение
Класс управления событиями фреймворка LOTIS. Предназначен для организации двусторонней коммуникации между клиентом и сервером без написания AJAX-кода. Серверная часть реализует регистрацию обработчиков событий, генерацию клиентского JavaScript-кода, обработку входящих POST-запросов и выполнение бизнес-логики. Клиентская часть обеспечивает отправку данных на сервер, сериализацию FormData, управление глобальными переменными и обработку ответов. Класс поддерживает систему хуков жизненного цикла событий (`check`, `before`, `on`), асинхронный и синхронный режимы работы, а также глобальную регистрацию всех экземпляров событий для маршрутизации запросов.

### Структура работы

1. **Клиентская часть:**
   * Формирование запроса
   * Отправка данных
   * Обработка ответа

2. **Серверная часть:**
   * Проверка условий
   * Выполнение логики
   * Возврат результата

## Свойства (серверная часть, PHP)

| Свойство | Тип | Видимость | Описание |
|----------|-----|-----------|----------|
| **$async** | `bool` | `public` | Флаг асинхронного режима. Если `true` — запросы выполняются асинхронно через `fetch`/`XMLHttpRequest`. Если `false` — синхронно. По умолчанию `true`. |
| **$eventresult** | `mixed` | `public` | Результат выполнения последнего обработанного события. Заполняется при вызове метода `join()`. |
| **$checktrue** | `bool` | `public` | Флаг успешности проверки хука `check`. Устанавливается в `true` перед выполнением события, может быть изменён хуком проверки. |
| **$checkerror** | `string` | `public` | Сообщение об ошибке проверки. Заполняется, если хук `check` возвращает не `true`. |
| **$all** | `static array` | `public static` | Статический реестр всех созданных объектов Events по их ID. Используется для маршрутизации входящих POST-запросов к правильному обработчику. |

### Наследуемые свойства

| Свойство | Тип | Видимость | Описание |
|----------|-----|-----------|----------|
| **$id** | `string` | `public` | Наследуется от `Construct`. Уникальный идентификатор объекта Events, используется для связи клиент-сервер. |
| **$type** | `string` | `public` | Наследуется от `Quark`. Устанавливается в `'Events'` в конструкторе. Определяет тип объекта для фильтрации и обработки. |
| **$storage** | `array` | `public` | Наследуется от `Quark`. Хранилище содержит секции `'childs'` (обработчики событий), `'errors'` (обработчики ошибок). |

## Объект ltsEvents (клиентская часть, JS)

| Свойство | Тип | Описание |
|----------|-----|----------|
| **async** | `bool` | Флаг асинхронного режима. Устанавливается сервером при компиляции. По умолчанию `true`. |
| **script** | `string` | Путь к скрипту для отправки POST-запросов. Заполняется автоматически при компиляции событий. |
| **post** | `function` | Метод отправки события на сервер. Принимает имя события, ID объекта и аргументы. |

## Методы

### Констируктор

| Метод | Параметры | Возвращает | Описание |
|-------|-----------|------------|----------|
| **__construct** | `string $id = ''` | — | Конструктор класса. Вызывает родительский конструктор `Quark`, устанавливает `$type` в `'Events'`, `$async` в `true`. Регистрирует объект в статическом реестре `Events::$all` по ID. |

### Программирование событий

| Метод | Параметры | Возвращает | Описание |
|-------|-----------|------------|----------|
| **client** | `string $name`, `string $body` | `$this` | Добавляет клиентский обработчик результата выполнения события на сервере. Принимает заголовок события и JavaScript-код обработчика. Код добавляется в JS-объект через метод `js()->add()`. Поддерживает fluent interface. `$name`, заголовок события, может содержать в скобках список параметров через запятую, как при обычном определении заголовка JS-функции. |
| **server** | `string $name`, `callable $body` | `$this` | Добавляет серверный обработчик события. Принимает имя события и PHP-функцию/замыкание. Если имя содержит параметры в скобках `name(params)` — извлекает только имя. Добавляет обработчик в секцию `'childs'`. Поддерживает fluent interface. Функция $body должна иметь единственный аргумент, в котором передается массив параметров, указанных при определении заголовка события в методе `client`. Даже если событие определено без параметров, указание аргумента обязательно. Метод должен возвращать результат, который будет передан на клиент в качестве ответа выполнения события |
| **error** | `string $name`, `string $body` | `$this` | Добавляет обработчик ошибок для события. Принимает имя события и JavaScript-код обработчика ошибки. Если код не начинается с `function` — оборачивает в функцию `function (error) { ... }`. Сохраняет в секции `'errors'` с ключом `{$this->id}_{$_name}`. |
| **add** | `string $name`, `callable $value = null` | `$this` | Переопределённый метод добавления потомка. Если передано замыкание — обрабатывает синтаксис имени с параметрами `name(params)`. Вызывает родительский `Quark::add()`. Поддерживает fluent interface. Метод нужно использовать для определения серверных хуков событий. В качестве имени хука используется имя события с префиксом `check`, `before` или `on`. Хуки, как и серверный обработчик события, принимает единственный аргумент, в котором передается массив принятых с клиента при вызове события параметров. Хук `before` предназначен для обработки и дополнения массива аргументов. Он должен возвращать массив, который будет передан основному обработчику события в качестве аргумента. Даже если не требуется преобразование массива аргументов, хук `before` должен вернуть этот массив в качестве результата работы. Хук `on`, кроме массива аргументов, вторым параметром принимает результат выполнения основного серверного обработчика события. Он предназначен для преобразования результата и должен везвратить его после своего выполнения.  |

### Серверная обработка запросов 

| Метод | Параметры | Возвращает | Описание |
|-------|-----------|------------|----------|
| **eventjoin** | `string $eventname`, `callable $event`, `array $args` | `mixed/false` | Внутренний метод выполнения серверного обработчика события с хуками. Последовательно вызывает: хук `check` (если есть — проверяет возврат `true`), хук `before` (может модифицировать аргументы), основной обработчик `$event`, хук `on` (получает результат). Возвращает результат выполнения или `false` при неудачной проверке. |
| **join** | `string $id_eventname` | `bool` | Обработка входящего POST-запроса. Извлекает имя события из `$id_eventname` (после `id_`). Парсит `$_POST` и `$_FILES` в массив аргументов, декодирует JSON-поля по ключам из `__ltsJSONKeys`. Вызывает `eventjoin()` для выполнения обработчика. Сохраняет результат в `$eventresult`. Возвращает `true` при успехе, `false` при неудачной проверке или отсутствии обработчика. |
| **build** | — | `void` | Основной статический метод обработки глобального POST-запроса событий. Вызывается автоматически объектом класса `Space` при наличии `__ltsEvent` в `$_POST`. Извлекает имя события и ID объекта из POST-данных. Находит объект в реестре `Events::$all`, вызывает `join()`. Собирает результаты всех переменных `Vars::$all` в `__ltsVars`. Выводит JSON-ответ и завершает выполнение скрипта через `exit()`. Хотя метод вызывается автоматически при построении страницы, можно программно отделить секцию серверной обработки событий от кода построения части приложения, не участвующего в событиях. Для этого в коде программы необходимо явно вызвать `LTS::Events()->build()` |

### Жизненный цикл

| Метод | Параметры | Возвращает | Описание |
|-------|-----------|------------|----------|
| **compile** | — | — | Генерация клиентского JavaScript-кода для всех событий. Создаёт объект `JS` с областью `'ready'`. Для каждого обработчика генерирует: функции хуков (`check`, `before`, `on`), основную функцию события с обработкой аргументов, FormData-сериализацию, POST-запрос через `ltsEvents.post()`, обработчики ответов и ошибок. Устанавливает `$this->js()->type` в `'none'` после компиляции. |

## Клиентские методы, доступные через LTS-объект

| Метод | Параметры | Возвращает | Описание |
|-------|-----------|------------|----------|
| **LTS(events).{eventname}** | `...args` | — | Метод вызова события на клиенте. Генерируется автоматически при компиляции Events.php. Принимает аргументы события, вызывает `ltsEvents.post()` с правильными параметрами. После получения ответа вызывает клиентский обработчик через `ltsEvents.__event{id}_{eventname}()`. |
| **LTS(events).check{eventname}** | `...args` | `bool` | Хук-валидатор события. Генерируется при компиляции если определён на сервере. Если возвращает `false` — событие не отправляется на сервер. |
| **LTS(events).before{eventname}** | `...args` | `mixed` | Хук предобработки события. Генерируется при компиляции если определён на сервере. Может модифицировать аргументы перед отправкой. Если возвращает FormData — используется как есть, если объект — поля добавляются к аргументам. |
| **LTS(events).on{eventname}** | `...args`, `result` | — | Хук постобработки события. Генерируется при компиляции если определён на сервере. Вызывается после получения ответа от сервера с результатом выполнения. |

### Вспомогательные клиентские методы

| Метод | Параметры | Возвращает | Описание |
|-------|-----------|------------|----------|
| **ltsEvents.post** | `string name`, `string id`, `mixed args` | — | Отправляет событие на сервер. Формирует FormData с полями `__ltsEvent`, `__ltsEventId`, `__ltsVars`, `__ltsJSONKeys`. Сериализует объекты и массивы в JSON, файлы добавляет как есть. Вызывает `LTS.post()` для отправки запроса. Обработчики ответов устанавливаются при компиляции в Events.php. |
| **LTS.request** | `string name`, `function func` | — | Регистрирует обработчик успешного ответа для события. Вызывается внутри сгенерированного кода при компиляции. Принимает данные ответа, обновляет `LTS.vars()` из `__ltsVars`, вызывает клиентский обработчик события. |
| **LTS.error** | `string name`, `function func` | — | Регистрирует обработчик ошибки для события. Вызывается внутри сгенерированного кода при компиляции или через метод `error()` на сервере. Принимает объект ошибки, выводит в консоль или выполняет пользовательский код. |

## Пример использования

```php
// Создание объекта событий
$events = LTS::Events('myEvents');

// Регистрация серверного события
$events->server('calculate', function($args) {
    return $args['a'] + $args['b'];
});

// Регистрация клиентского обработчика результата
$events->client('calculate(a, b)', function(result) {
    alert('Результат: ' + result);
});

// Регистрация клиентского хака
$events->method('checkcalculate(a, b)',
<<<JS
    if(typeof a != 'number' || typeof b != 'number')
    {
        alert('Введите числа!');
        return false;  
    }
    return true;
JS
);

// Регистрация серверного хака
$events->add('oncalculate', function ($args, $result) {
    if($args['a'] == $args['b'])
        return $result * 2;
    else
        return $result;
});

// Пример вызова события
$button = LTS::Button()
    ->capt('Сложим 5 и 3')
    ->click(
<<<JS
        LTS(events).calculate(5, 3);
JS
);
```

## Примечания
*   Класс расширяет `Quark`, поэтому поддерживает все методы управления потомками: `add()`, `get()`, `set()`, `del()`, `getchilds()` и др.
*   Класс расширяет `Construct`, поэтому поддерживает все хуки жизненного цикла: `$check`, `$before`, `$on`, `$checkchilds`, `$beforechilds`, `$onchilds`.
*   Все объекты Events автоматически регистрируются в статическом реестре `Events::$all` — это необходимо для маршрутизации POST-запросов к правильному обработчику.
*   Метод `build()` должен быть вызван до рендеринга страницы — обычно это делается в точке входа фреймворка через `Space::metadata()` или `Space::build()`.
*   Хуки жизненного цикла событий: `check{eventname}` (валидация), `before{eventname}` (предобработка аргументов), `on{eventname}` (постобработка результата).
*   При асинхронном режиме (`$async == true`) запросы выполняются через `fetch` или `XMLHttpRequest` без блокировки интерфейса.
*   При синхронном режиме (`$async == false`) запросы блокируют выполнение до получения ответа — используется редко, для специфических сценариев.
*   Обработчики ошибок регистрируются через метод `error()` и выполняются на клиенте при неудачном ответе сервера.
*   Метод `compile()` генерирует сложный JavaScript-код с обработкой FormData, JSON-сериализацией, хуками и обработчиками ответов — весь этот код прозрачен для разработчика.
*   Для событий с параметрами `name(params)` — параметры извлекаются из имени и используются для генерации сигнатуры функции на клиенте.
*   Аргументы события могут включать данные из `$_POST` и `$_FILES` — файлы автоматически добавляются в массив аргументов.
*   JSON-поля декодируются по списку ключей из `__ltsJSONKeys` — это позволяет передавать сложные структуры данных в POST-запросе.
*   Результат выполнения события возвращается в клиентский обработчик через поле `{eventname}` в JSON-ответе.
*   Глобальные переменные `Vars` автоматически синхронизируются между сервером и клиентом через поле `__ltsVars` в ответе.
*   Метод `client()` используется для регистрации обработчиков ответа на стороне клиента — код выполняется после получения успешного ответа от сервера.
*   Метод `server()` используется для регистрации обработчиков на стороне сервера — функция вызывается при получении POST-запроса с именем события.
*   Класс не реализует методы `shine()` и `childs()` — они наследуются от `Quark` без изменений, так как Events не создаёт DOM-элементов напрямую.
*   Объекты Events не имеют визуального представления — они генерируют только JavaScript-код и обрабатывают POST-запросы.
*   Для вызова события на клиенте используется синтаксис `LTS.get('{id}').{eventname}(args)` — функция создаётся автоматически при компиляции. Также возможен вызов `LTS(var).{eventname}(args)`, где var - PHP-переменная, ссылающаяся на экземпляр класса `Events`
*   Клиентская часть (Events.js) содержит объект `ltsEvents` с методом `post()` — он отвечает за формирование FormData и отправку запроса через `LTS.post()`.
*   Серверная часть (Events.php) генерирует клиентский код при компиляции — все обработчики, хуки и настройки записываются в JS-объект.
*   При обработке событий через `Events::build()` все объекты Vars автоматически вызывают `store()` — изменения сохраняются в сессию перед отправкой ответа клиенту.
*   Для работы клиентской части требуется наличие файла `Events.js`, который подключается автоматически через пространство имён Space.
*   Метод `ltsEvents.post()` автоматически определяет тип данных — если аргументы содержат объекты или массивы, они сериализуются в JSON с префиксом `<JSON>`.
*   Файлы передаются через FormData без сериализации — `ltsEvents.post()` проверяет instanceof File и добавляет как есть.
*   Обработчики ответов устанавливаются через `LTS.request()` внутри сгенерированного кода — разработчик не вызывает их явно.
*   Ошибки обработки событий логируются в консоль браузера через `console.error()` если не задан пользовательский обработчик через `error()`.
*   Синхронизация переменных `Vars` происходит автоматически — все изменения на сервере передаются клиенту в поле `__ltsVars` JSON-ответа.
*   Для отладки событий можно использовать `console.log()` в клиентских обработчиках и `Logger` на сервере.
*   Класс поддерживает множественные события в одном объекте — каждое событие имеет уникальный обработчик и хуки.
*   При компиляции Events.php создаёт JS-объект с областью `'ready'` — код выполняется после загрузки DOM.
*   Флаг `$async` устанавливается в конструкторе — может быть изменён до вызова `compile()` для смены режима работы.
